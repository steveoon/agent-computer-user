# Duliday 品牌别名查询接口需求文档

## 背景

我方系统在处理招聘对话时，需要从求职者的自然语言消息中识别品牌关键词（如用户说"想去KFC"，系统需识别为"肯德基"）。

当前品牌别名数据**硬编码在前端代码中**，每次新增/修改品牌都需要发版。希望 Duliday 提供一个品牌别名查询接口，实现别名数据的动态获取。

### 我方使用方式

```
接口返回全量品牌别名 → 缓存 5 分钟
                          ↓
用户消息: "想去KFC上班"  → 遍历别名字典匹配 → 命中 "KFC" → 识别为 "肯德基"
```

核心诉求：**一次性获取所有品牌及其别名**，用于文本关键词匹配，不需要按组织/项目筛选。

## 接口定义

### 请求

```
GET /api/brand/alias/list
```

无必填参数。返回当前所有活跃品牌及其别名。

### 响应

```jsonc
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "brandId": 101,
      "brandName": "肯德基", // 品牌标准名称（我方用此名称作为匹配结果）
      "projectId": 1001, // 品牌所属项目 ID（即原来的 organizationId）
      "aliases": ["KFC", "kfc", "肯记"], // 别名列表（含英文、缩写、口语化称呼等）
    },
    {
      "brandId": 101,
      "brandName": "北京肯德基", // 区域品牌作为独立条目
      "projectId": 1043,
      "aliases": ["北京KFC", "北京kfc"],
    },
    {
      "brandId": 102,
      "brandName": "必胜客",
      "projectId": 1002,
      "aliases": ["Pizza Hut", "PizzaHut"],
    },
    {
      "brandId": 103,
      "brandName": "奥乐齐",
      "projectId": 1003,
      "aliases": ["ALDI", "Aldi"],
    },
    {
      "brandId": 104,
      "brandName": "大米先生",
      "projectId": 1004,
      "aliases": [], // 无别名也返回空数组
    },
  ],
}
```

### 字段说明

| 字段        | 类型     | 必填 | 说明                                                                                               |
| ----------- | -------- | ---- | -------------------------------------------------------------------------------------------------- |
| `brandId`   | number   | 是   | 品牌唯一 ID                                                                                        |
| `brandName` | string   | 是   | 品牌标准名称，我方用此名称作为识别结果和内部 key                                                   |
| `projectId` | number   | 是   | 品牌所属项目 ID（对应我方系统中的 organizationId，用于同步数据时关联）                             |
| `aliases`   | string[] | 是   | 别名列表，空数组表示无别名。**不需要包含 `brandName` 本身**，我方会自动将 `brandName` 加入匹配词表 |

### 别名覆盖范围

每个品牌的 `aliases` 应尽量覆盖以下变体：

| 类型         | 示例（以肯德基为例） |
| ------------ | -------------------- |
| 英文全称     | KFC                  |
| 英文小写     | kfc                  |
| 口语化简称   | 肯记                 |
| 常见拼写变体 | K.F.C                |

不需要包含的：

- 品牌标准名称本身（`brandName` 已有）

### 区域品牌处理

区域品牌（如"北京肯德基"、"大连肯德基"）**直接作为独立条目平铺**在 `data` 数组中即可，不需要嵌套在父品牌下。我方系统中每个区域品牌都是独立的业务实体。

## 调用场景

- **调用频率**：低频，我方会缓存 5 分钟，预计单实例每小时调用 ≤ 12 次
- **数据量**：预计总品牌数 < 100，单次响应 < 50KB
- **调用时机**：服务端调用（Node.js），非用户请求的关键路径
- **接口鉴权**：沿用现有的 `Duliday-Token` Header

## 其他说明

- 如果 `aliases` 字段的维护成本较高，也可以全部返回空数组，我方会用 `brandName` 本身做基础匹配（降级但可用）
- 响应结构灵活，只要包含 `brandName` + `aliases` 两个核心字段即可，其他字段随意
